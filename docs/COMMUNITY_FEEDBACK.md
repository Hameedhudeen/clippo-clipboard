# Community Feedback Policy

Clippo should keep feedback focused while the project is pre-alpha.

## Use Issues For

- Reproducible bugs.
- Platform-specific behavior reports.
- Workflow parity gaps.
- Accessibility problems.
- Documentation corrections.
- Small, well-scoped feature requests.

## Use Pull Requests For

- Fixes linked to an issue or checklist item.
- Documentation updates.
- Tests that capture known gaps.
- Platform implementation behind the existing architecture.

## Feature Requests

Feature requests should explain:

- The workflow problem.
- The affected operating system.
- Whether the request is needed for workflow parity or is a post-v1 enhancement.
- Any privacy, permission, or performance tradeoff.

## Discussions

GitHub Discussions can be enabled later if issue traffic becomes noisy. Until then, broad questions should use issues with the `question` or `help wanted` label so decisions remain visible to contributors.
